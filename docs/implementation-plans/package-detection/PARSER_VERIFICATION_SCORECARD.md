# Package Parser Verification Scorecard

This scorecard tracks **end-to-end parser verification targets** for implemented package and package-adjacent parser families.

Unlike [`PARSER_PLAN.md`](PARSER_PLAN.md), which is now a historical implementation/backlog document, this file is the maintained checklist for deciding **what to run through** `compare-outputs` and for recording **which parser families have already been compared against ScanCode**. Detailed benchmark examples and timing snapshots live in [`../../BENCHMARKS.md`](../../BENCHMARKS.md).

The ranked table below intentionally follows the implemented ecosystem rows in [`PARSER_PLAN.md`](PARSER_PLAN.md), while also allowing explicit **cross-cutting non-parser rows** when a target is valuable as broad ecosystem coverage but does not map to a single parser family.

## Required comparison methodology

Use the `xtask` compare workflow for parser verification. Do **not** substitute ad hoc manual scanner invocations for the main parity check.

Repository-backed target:

```bash
cargo run --manifest-path xtask/Cargo.toml --bin compare-outputs -- --repo-url https://github.com/org/repo.git --repo-ref <ref> --profile common
```

Artifact/rootfs-backed target:

```bash
cargo run --manifest-path xtask/Cargo.toml --bin compare-outputs -- --target-path /path/to/local/target --profile common
```

Method rules:

- Run parser-family comparisons with `--profile common`, not `--profile packages`, so package extraction is always evaluated alongside installed-package DB coverage, license, copyright, email, and URL detection.
- Treat any “more output” from either scanner as a claim to verify, not as proof by itself. Additional licenses, copyrights, holders, authors, emails, or URLs only count as improvements when the scanned file text actually supports them.
- Fix issues where **ScanCode is better than Provenant**.
- When scanners disagree, inspect the underlying file text enough to decide whether the extra or missing finding is justified. If both scanners agree, lighter spot-checking is usually sufficient.
- Do **not** mark a row `🟢 Verified` while any ScanCode-better deltas remain unresolved for that target. Treat `comparison_status: potential_regressions_detected` as a triage-required signal, not an automatic failure: each such delta must be reviewed and either fixed or documented as not actually worse before the row becomes `🟢 Verified`.
- Keep and document differences where **Provenant is better than ScanCode**.
- Do **not** treat normalization improvements as regressions when Provenant is more correct, for example preserving `René` instead of degrading to `Rene`. Parity is the bottom line, not the upper limit.
- When a fix touches shared detection logic, run the relevant regression suites for that subsystem. For example, copyright-detection changes should rerun the copyright goldens; license-detection changes should rerun the license goldens; parser behavior fixes should rerun the narrow parser tests and any affected compare targets.
- Any fixed regression should gain adequate automated coverage. Add focused tests, including golden tests when that is the most durable fit.
- Keep detailed diff analysis in PR descriptions, CI logs, and saved `.provenant/compare-runs/` artifacts rather than bloating this checklist.

## Status model

- `⚪ Planned` — candidate targets are listed, or compare runs exist, but verification work is not complete yet.
- `🟢 Verified` — at least one representative compare target has been run, and any ScanCode-better findings for that target were fixed or triaged as not actually worse.

## Scope notes before using the ranked table

- `npm / yarn / pnpm (+ Bun)` is one consolidated implemented family from [`PARSER_PLAN.md`](PARSER_PLAN.md). Bun was not omitted; it is intentionally tracked inside the npm-family row because the implementation consolidates Bun text lockfiles and legacy `bun.lockb` v2 compatibility into that family.
- `Python / PyPI` is also a consolidated family. It includes modern surfaces such as `uv.lock`, `pylock.toml`, installed `WHEEL`, and pip cache `origin.json`, even when a repo-level benchmark only exercises part of that surface area.
- There is **no generic `C++` parser family** in [`PARSER_PLAN.md`](PARSER_PLAN.md). Broad `C++` repositories therefore act as **cross-cutting references** for families such as Autotools, Conan, vcpkg, Bazel, and Buck rather than as a standalone parser row.
- Large **polyglot/vendored monorepos** are also worth tracking separately. They often reveal interaction bugs across parser families, package-adjacent metadata, and common-profile text detection that are easy to miss in smaller ecosystem-focused repos.
- Unpacked **rootfs and shipped-artifact snapshots** deserve cross-cutting tracking too. They simultaneously stress Linux distro metadata, package DB/archive handling, README/package-adjacent files, and common-profile detection in ways that source-only repos do not.
- Some rows are best exercised with **artifact or rootfs targets** instead of source repositories: Alpine archive/DB runs, Debian/RPM archive or installed-database runs, FreeBSD compact manifests, Linux distro `os-release`, Windows Update manifests, and similar shipped-artifact surfaces.
- `Structured metadata`, `README`, `Git submodules`, `Docker`, `Linux Distro`, `Windows Update`, and `misc.py` recognizers are **package-adjacent** rather than classic package-manager families. They still belong here because they interact with end-to-end scan results.
- Implemented beyond-parity parser families that are documented elsewhere but are **not** top-level rows in `PARSER_PLAN.md` stay out of this scorecard for now. That includes items such as Helm, Meson, and Pixi.

## How to maintain this file

When a parser family is verified with `compare-outputs`:

1. Run the `xtask` compare command with `--profile common`.
2. Fix the places where ScanCode is better.
3. Keep and document the places where Provenant is better.
4. Update the row directly from `⚪ Planned` to `🟢 Verified` only after no unresolved ScanCode-better deltas remain for that target; keep runs that still need fixes or triage as `⚪ Planned`.
5. Add or update focused automated tests for any regression fixed during that work, including golden tests when that is the right durable coverage.
6. After the row is fully validated, add or update the corresponding verification/benchmark entry in [`../../BENCHMARKS.md`](../../BENCHMARKS.md).

If a family is best tested on extracted package artifacts instead of Git repositories, keep the candidate as an artifact-style target here and use `compare-outputs --target-path` when the actual run happens.
